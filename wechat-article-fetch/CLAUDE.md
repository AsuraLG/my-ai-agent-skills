# CLAUDE.md

## 项目概况

这是一个 Claude skill 项目，用于搜索并抓取微信公众号文章的原始内容。

项目当前目标不是做通用网页搜索，也不是做公众号后台管理，而是：
- 通过搜狗微信搜索抓取微信公众号文章结果
- 解析真实的 `mp.weixin.qq.com` 原始文章链接
- 抽取文章标题、公众号名、发布时间、作者、正文纯文本
- 将抓取结果交给 Claude 按用户意图输出

当前产品定位已经明确：**这是一个“原始信息抓取 skill”，不是默认摘要 skill。**

---

## 当前目录结构

- `SKILL.md`
  - skill 的定义文件
  - 负责说明何时触发这个 skill、如何调用脚本、默认回复策略、失败处理方式

- `scripts/fetch.py`
  - 核心抓取脚本
  - 当前负责：
    - 请求搜狗微信搜索第一页
    - 提取 `/link?...` 跳转链接
    - 解析真实微信公众号文章 URL
    - 请求文章 HTML
    - 提取标题、账号名、发布时间、作者、正文文本
    - 输出 JSON

- `tests/test_fetch.py`
  - 回归测试文件
  - 当前使用 `unittest`
  - 用于保护链接提取、正文提取、元信息提取、代理配置、limit 归一化、正文清洗等行为

## 当前行为约定

### 搜索范围
- 只抓取搜狗微信搜索**第一页**
- 当前**不做分页**
- 不要重新引入分页逻辑，除非需求发生明确变化

### 参数约定
- 当前只保留一个业务参数：`--limit`
- 默认值：`10`
- 最大值：`10`
- 如果传入超过 10 的值，会被截断到 10
- 如果传入小于等于 0 的值，会抛出错误

### 返回内容定位
脚本返回的是抓取结果 JSON，包含：
- `title`
- `url`
- `account_name`
- `publish_time`
- `author`
- `content`

其中 `content` 是提取后的正文纯文本，而不是 HTML。

### assistant 默认回复策略
这个约定很重要，后续不要轻易改掉：
- 用户没有特殊说明时：**默认返回抓取到的文章标题 + 原始微信链接**
- 用户明确要求“摘要 / 总结 / 提炼”时：再基于 `content` 做摘要
- 用户明确要求“正文”时：再展开正文文本

也就是说：**默认行为是抓原始信息，不是主动帮用户总结。**

### 代理约定
- 代理默认关闭
- 只有在设置环境变量 `WECHAT_ARTICLE_FETCH_PROXY` 时，脚本才会同时配置 HTTP/HTTPS 代理
- 不要恢复成硬编码代理

### 失败处理约定
- 搜索结果页没解析出可用文章时：返回空结果，不编造内容
- 某篇文章跳转解析失败：跳过该条，继续处理其他结果
- 某篇文章正文抓取失败：仍返回标题/链接等基础结果，`content` 允许为空字符串

## 当前已完成进展

### 1. 抓取主流程已完成
当前脚本已经完成以下主链路：
1. 请求搜狗微信搜索页
2. 提取候选跳转链接
3. 解析真实 `mp.weixin.qq.com` URL
4. 请求公众号文章 HTML
5. 提取标题、元信息、正文
6. 输出 JSON

### 2. 元信息提取已完成
当前已支持提取：
- 标题
- 公众号名称
- 发布时间
- 作者

发布时间支持两种情况：
- 如果抓到 Unix 时间戳，则格式化为北京时间 `YYYY-MM-DD HH:MM:SS`
- 如果抓到的是可读文本，则原样保留

### 3. 正文纯文本提取已完成
当前正文提取逻辑已经支持：
- 定位 `id="js_content"`
- 去掉 `script` / `style`
- 将 `<br>` 转成换行
- 将常见块级标签结束位置转换为换行
- 去掉其余 HTML 标签
- 归一化空白字符

### 4. 正文噪音清洗已做过多轮增强
当前已经加入了一批针对真实微信公众号文章尾部噪音的过滤规则，已覆盖的典型类型包括：
- 阅读原文
- 扫码关注 / 微信扫一扫关注
- 长按二维码关注
- 本内容为作者独立观点…
- 如对本稿件有异议或投诉…

- 21君荐读 / 往期推荐 / 期待关注

- 扫码关注 / 关注我们
- 出品丨 / 作者丨 / 编辑丨 / 微信统筹丨
- `End`

### 5. 可选代理改造已完成
最初抓取流程曾受代理配置影响，后续已经改成：
- 默认不走代理
- 通过环境变量启用代理

这是一个明确确认过的产品约定。

### 6. 回归测试已建立
当前测试已覆盖：
- 搜索页链接提取
- 正文段落换行保留
- 常见公众号噪音过滤
- `<br>` 换行行为
- 发布时间格式化
- 元信息提取
- 默认无代理
- 环境变量代理生效
- `limit` 上限与非法值处理

## 当前已知问题 / 暂停事项

这些问题已经识别，但目前是**已知待处理项**，不是当前阻塞点：

### 1. 部分文章仍可能存在残余尾部噪音
真实样本里仍观察到一些未完全清洗掉的内容，例如：
- 文件下载提示
- 知识岛 / 社群 / 扫码加入类文案
- 投稿 / 商务合作类文案
- 图片来源 / 侵删声明
- 关注卡片、资料领取、交流群引导文案

当前这些优化已**暂时暂停**，除非后续重新开启这一轮清洗工作。

### 2. 少量文章正文可能为空
即使标题和原始链接抓到了，某些文章依然可能出现 `content == ""` 的情况。

这在当前设计里是允许的，属于 fail-soft 行为，不应因为单篇正文失败而让整个搜索结果失败。

### 3. 仓库目前没有可参考的提交历史
当前仓库状态显示还没有正式 commit 历史，因此未来维护时：
- 不要假设可以通过 `git log` 追溯设计演化
- 这份 `CLAUDE.md` 会承担一部分项目记忆的角色

## 维护建议

### 1. 优先保持“原始抓取 skill”的定位
如果以后要继续改这个项目，首先要守住这条边界：
- 这个 skill 的核心是**抓取微信公众号原始内容**
- 默认输出应偏向原始结果（标题、链接、必要元信息）
- 不要把默认行为改成“自动摘要优先”

### 2. 文档与脚本行为必须保持一致
如果修改了：
- 参数
- 默认返回策略
- 代理行为
- 输出 JSON 结构
- 失败处理方式

则必须同步更新 `SKILL.md`，必要时也更新本文件。

### 3. 正文清洗优先走测试驱动
如果后续继续扩充正文去噪规则，建议遵循当前已经验证过的方式：
1. 先从真实抓取样本里抽出最小失败片段
2. 在 `tests/test_fetch.py` 里补回归测试
3. 再扩展 `NOISE_PATTERNS` 或清洗逻辑
4. 确认旧行为不被破坏

不要先堆规则再补测试，否则很容易误伤正常正文。

### 4. 不要轻易扩大抓取范围
当前设计有意保持简单：
- 只抓第一页
- 只保留一个 `limit`
- 不做更复杂的搜索控制

除非需求明确变化，否则不要主动引入：
- 分页
- 更多搜索参数
- 更复杂的排序或过滤逻辑

### 5. 保持 fail-soft，而不是单点失败中断全部结果
当前实现更偏向“尽量返回可用结果”。

后续维护时应继续保持：
- 某篇文章失败，不要拖垮整个搜索
- 能返回标题和链接时，尽量返回
- 不要为了追求单篇完美而降低整体可用性

---

## 当前推荐维护顺序

如果后续继续迭代，推荐优先级如下：
1. 继续基于真实样本补正文尾部噪音清洗规则
2. 补更多失败场景测试（例如文章抓取失败、跳转解析失败）
3. 在不破坏简单性的前提下，提升正文提取稳定性
4. 最后才考虑新增能力
